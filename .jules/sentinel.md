## 2024-05-18 - [Razorpay Webhook Timing Attack Vulnerability]
**Vulnerability:** A timing attack vulnerability was identified in the `src/app/api/webhooks/razorpay/route.ts` file, where a standard string equality check (`!==`) was used to compare the computed HMAC signature against the incoming Razorpay webhook signature. This could allow an attacker to potentially guess the correct signature by measuring the response time, leading to forged webhook payloads and unauthorized order status updates (e.g., marking unpaid orders as paid).
**Learning:** Webhook signature verification in this codebase (and generally) must use a constant-time comparison to prevent timing attacks. Additionally, relying purely on string comparisons for security-critical checks is insufficient. Also learned that `pnpm install` generates a `pnpm-lock.yaml` which should be reverted if it's not the explicit intention to commit it in a security fix.
**Prevention:** Always use `crypto.timingSafeEqual` for comparing cryptographic hashes, signatures, or secrets. Ensure that inputs to `timingSafeEqual` are converted to `Buffer` objects, apply fallbacks for potentially undefined inputs (e.g., `Buffer.from(signature || '')`), and perform a length check before the comparison to prevent errors. Ensure auto-generated files like `pnpm-lock.yaml` are reverted when fixing localized security issues.
